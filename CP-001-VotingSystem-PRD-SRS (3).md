# PART A. PRD (Product Requirements Document)

## 1. 배경 및 목적

블록체인의 핵심 가치인 **투명성·불변성·검증가능성**을 가장 직관적으로 체험할 수 있는 도메인이 "투표"이다. 본 시스템은 수업 실습을 위해 다음 학습 목표를 달성한다.

- **온체인 / 오프체인 분리 설계** — 어떤 데이터가 블록체인에 올라가야 하고 어떤 데이터는 일반 서버에서 관리되어야 하는지 직접 결정·구현
- **메타마스크 기반 사용자 인증** — 별도 회원가입 없이 지갑 주소로 본인 확인 및 투표
- **스마트 컨트랙트 설계 패턴** — 권한 제어(`onlyOwner`), 상태 머신(IDLE → ACTIVE → ENDED), 이벤트 기반 UI 갱신
- **실시간 결과 시각화** — 한국 개표 방송(KBS, MBC, SBS 출구조사 그래픽) 스타일의 가시성 높은 UX 제공

## 2. 제품 개요 (Elevator Pitch)

> "교수자가 후보를 등록하고 투표를 개시하면, 학생들은 자신의 메타마스크 지갑으로 단 한 번 투표할 수 있다. 투표가 진행되는 동안 모든 참여자는 실시간으로 갱신되는 개표 방송 스타일 대시보드를 통해 득표 현황을 지켜보고, 종료 시점에 최종 결과가 확정 공표된다. 모든 표는 세폴리아 테스트넷에 영구 기록된다."

## 3. 사용자 페르소나

| 페르소나 | 설명 | 권한 | 핵심 행동 |
|----------|------|------|-----------|
| 관리자(Admin) | 스마트 컨트랙트 배포자. 강사 또는 조교 1명 | 후보 등록/삭제, 투표 시작/종료, 본인도 투표 가능 | 컨트랙트 배포 → 후보 등록 → 투표 개시 → (필요 시) 조기 종료 |
| 유권자(Voter) | 세폴리아 지갑을 가진 모든 사용자 (수강생) | 투표 1회, 결과 조회 | 지갑 연결 → 후보 확인 → 투표 → 결과 확인 |
| 참관인(Observer) | 지갑이 없거나 투표를 원치 않는 사용자 | 결과 조회만 가능 | 결과 대시보드 열람 |

## 4. 핵심 사용자 시나리오 (User Journey)

### 시나리오 A. 관리자가 투표를 개설한다

1. 관리자가 웹 앱에 접속하여 메타마스크로 연결한다.
2. 현재 동작 중인 투표가 없거나 직전 투표가 `ENDED` 상태이면, 메인 화면(S1-NONE)에 **"+ 새 투표 만들기"** 버튼이 활성화된다.
3. 관리자가 버튼을 클릭하면 `/new` 화면(S-N)으로 이동하며, 사전 점검(`POST /votings/precheck`) 결과 `canDeploy=true` 일 때 배포 버튼이 활성화된다.
4. 투표 제목을 입력하고 **"컨트랙트 배포하기"** 버튼을 클릭하면 메타마스크 서명 팝업이 열리고, 승인 시 `Voting.sol` 컨트랙트가 세폴리아에 배포된다.
5. 배포 완료 후 프론트엔드가 트랜잭션 receipt에서 컨트랙트 주소를 읽고 `POST /votings`로 백엔드에 등록한다.
6. 배포 완료 후 후보자 등록 화면(`/admin/candidates`)으로 자동 이동하며, 이름·사진을 입력하여 최소 2명 이상 등록한다.
7. "투표 시작" 화면에서 종료 일시를 선택하고 트랜잭션을 서명한다.
8. 컨트랙트 상태가 `ACTIVE`로 전환되고, 모든 사용자에게 투표 페이지가 열린다.

### 시나리오 B. 유권자가 투표한다

1. 유권자가 웹 앱에 접속한다.
2. "지갑 연결" 버튼으로 메타마스크를 연결한다 (네트워크가 세폴리아가 아니면 자동 전환 요청).
3. 후보자 카드 목록을 확인한다 → 한 명을 선택한다.
4. "투표하기" 버튼 클릭 → 메타마스크 서명 팝업 → 가스비 확인 후 승인.
5. 트랜잭션이 채굴되면 "투표 완료" 화면으로 전환되고, 본인의 투표 내역(트랜잭션 해시)이 표시된다.
6. 같은 지갑으로 재진입 시 투표 버튼은 비활성화되고 "이미 투표하셨습니다" 메시지가 표시된다.

### 시나리오 C. 모두가 결과를 지켜본다

1. 투표 진행 중에는 메인 대시보드에 **실시간 막대 차트 + 도넛 차트 + 누적 카운터**가 5초 주기로 갱신된다.
2. 종료 시각에 도달하거나 관리자가 수동 종료하면 화면이 "최종 결과" 모드로 전환된다.
3. 1위 후보의 카드가 강조 애니메이션과 함께 "당선" 배지로 표시된다.

## 7. 범위 (Scope)

### In Scope

- 단일 선거 운영 (한 컨트랙트 = 한 투표)
- **웹 UI에서 컨트랙트 배포** (Hardhat CLI 불필요)
- **배포 잠금**: 투표가 `IDLE` 또는 `ACTIVE`이면 재배포 불가, `ENDED` 후 재배포 가능
- 공개 투표 (실시간 득표 공개)
- 누구나 투표 가능 (지갑만 있으면 됨)
- 후보자 사진 업로드 (오프체인 저장)

### Out of Scope

- 복수 선거 동시 운영
- 비밀 투표(commit-reveal)
- 화이트리스트/KYC
- 모바일 전용 UI
- 메인넷 배포
- 다국어 지원 (한국어 단일)
- 후보자별 SNS 링크, 공약 텍스트 등 부가 정보 (확장 여지로만 둠)

## 8. 가정 및 제약

- 모든 사용자는 **세폴리아 테스트 ETH**를 미리 보유하고 있다 (Faucet 안내는 별도 자료).
- 관리자는 **단 1명**(컨트랙트 배포 주소)이며, 양도/추가는 지원하지 않는다.
- 컨트랙트 배포는 Hardhat CLI 없이 **웹 앱 UI에서 직접** 수행한다. 별도 개발 환경 설치 불필요.
- 가장 최근에 배포된 컨트랙트가 현재 투표 기준이며, `ENDED` 상태에서만 새 컨트랙트를 배포할 수 있다.
- 후보자 본인의 동의·신원 검증은 본 시스템 범위 밖이다 (실습 목적).
- 사진 파일은 ≤ 2MB, JPG/PNG/WebP만 허용한다.

## 9. 시스템 아키텍처

### 9.1 전체 구성도

```
┌─────────────────────────────────────────────────────────┐
│                    Browser (Chrome)                      │
│                                                         │
│  ┌──────────────────────┐      ┌─────────────────────┐  │

         ▼                               ▼
┌──────────────────────┐    ┌───────────────────────────┐
│  Backend             │    │      Sepolia Testnet       │
│  (Node/Express       │    │                           │
│   + SQLite + Multer) │    │  ┌─────────────────────┐  │

│    onChainIndex 매핑  │    └───────────────────────────┘
│  - 컨트랙트 상태 조회 │              ▲

└──────────────────────┘
```

> ⚠️ 백엔드도 Sepolia RPC에 직접 연결해야 한다. `POST /votings/precheck`에서 현재 컨트랙트 상태(IDLE/ACTIVE/ENDED)를 체인에서 직접 읽어야 하며, 과거 투표 결과(`winner`, `totalVotes`) 집계 시에도 필요하다. 백엔드 환경변수에 `SEPOLIA_RPC_URL`과 `CONTRACT_ABI_PATH`가 반드시 포함되어야 한다.

### 9.2 책임 분리 (온체인 vs 오프체인)

| 데이터/로직 | 위치 | 사유 |
|-------------|------|------|
| 후보자 ID(uint), 득표수 | 온체인 | 투표 무결성의 핵심 |
| 투표 상태(IDLE/ACTIVE/ENDED), 종료 시각 | 온체인 | 시작/종료의 객관적 진실 |
| 투표자 주소 ↔ 투표 여부 mapping | 온체인 | 중복 방지의 강제 수단 |
| `Voted` 이벤트 (시간순 로그) | 온체인 | 감사 추적 |
| 후보자 이름(한글) | 오프체인 | 가스비, 변경 가능성, 인코딩 편의 |
| 후보자 사진 | 오프체인 | 가스비(이미지 KB→ETH) |
| 후보자 등록 시각, 부가 메타 | 오프체인 | 검증 불요 |
| 배포된 컨트랙트 주소 이력 | 오프체인 (백엔드 DB) | 현재 활성 컨트랙트 주소를 프론트에 제공하기 위한 레지스트리 |

## 10. 스마트 컨트랙트 명세 (`Voting.sol`)

### 10.1 상태 머신

```
         startVoting(endTime)              endVoting() | block.timestamp ≥ endTime
[ IDLE ] ──────────────────────► [ ACTIVE ] ──────────────────────────────────► [ ENDED ]
    ▲
    │ 후보자 등록/수정/삭제는 IDLE 상태에서만
```

> 본 시스템은 단일 선거 1회용이다. `ENDED` 이후 재시작 없음. 새 선거가 필요하면 컨트랙트 재배포.

### 10.2 데이터 구조

| 변수 | 타입 | 설명 |
|------|------|------|
| `owner` | `address` | 배포자 (Ownable) |
| `state` | `enum {IDLE, ACTIVE, ENDED}` | 현재 상태 |
| `endTime` | `uint256` | 종료 예정 unix timestamp |
| `candidateCount` | `uint256` | 유효 후보자 수 (삭제된 후보 제외) |
| `nextCandidateId` | `uint256` | 다음 후보자에게 부여할 ID (자동 증가, 삭제해도 재사용 안 함) |
| `votes[id]` | `mapping(uint256 ⇒ uint256)` | 후보자별 득표수 |
| `isActive[id]` | `mapping(uint256 ⇒ bool)` | 후보자 유효 여부 (삭제 시 false) |
| `hasVoted[addr]` | `mapping(address ⇒ bool)` | 중복 방지 |
| `votedFor[addr]` | `mapping(address ⇒ uint256)` | 본인 투표 이력 조회용 |

> 후보자 이름·사진은 컨트랙트에 저장하지 않는다. 백엔드 DB가 `candidateId` (컨트랙트의 인덱스)와 1:1 매핑.

### 10.3 함수 명세

| 함수 | 권한 | 상태 조건 | 설명 |
|------|------|-----------|------|
| `addCandidate()` | onlyOwner | IDLE | 후보자 1명 추가, `candidateCount++`. 새 ID는 `CandidateAdded(id)` 이벤트로 프론트엔드에 전달 (솔리디티 상태변경 함수의 반환값은 외부 호출자가 직접 읽을 수 없으므로 이벤트를 사용) |
| `removeCandidate(id)` | onlyOwner | IDLE | 해당 ID의 후보자 비활성화. **ID는 재사용하지 않으며 해당 슬롯은 무효(isActive=false)로 표시.** `candidateCount`는 유효 후보자 수를 추적하기 위해 감소. 프론트엔드·백엔드는 ID를 기준으로 조회하므로 빈 슬롯은 필터링하여 표시. |
| `startVoting(endTime)` | onlyOwner | IDLE & `candidateCount ≥ 2` & `endTime > now` | ACTIVE로 전환, `VotingStarted(endTime)` |
| `vote(id)` | 누구나 | ACTIVE & `now < endTime` & `!hasVoted[msg.sender]` & 유효 ID | 표 +1, `Voted(voter, id, timestamp)` |
| `endVoting()` | onlyOwner | ACTIVE | 강제 종료, `VotingEnded(reason="manual")` |
| `getResults()` | view, 누구나 | any | 유효한 후보자(`isActive[id]==true`)만 필터링하여 `(ids[], votes[])` 배열 반환. 삭제된 후보의 ID는 포함하지 않아 프론트에서 별도 처리 불필요 |
| `getState()` | view | any | 현재 상태(시각 기준 자동 계산: ACTIVE이지만 `now ≥ endTime`이면 ENDED로 응답) |
| `getMyVote()` | view | any | 호출자의 `(hasVoted, votedFor)` |

### 10.4 이벤트

```solidity
event CandidateAdded(uint256 indexed id);
event CandidateRemoved(uint256 indexed id);
event VotingStarted(uint256 endTime);
event VotingEnded(string reason);  // "manual" | "timeup"
event Voted(address indexed voter, uint256 indexed candidateId, uint256 timestamp);
```

### 10.5 에러 (custom error)

| 에러 | 발생 조건 |
|------|-----------|
| `NotOwner()` | 관리자 전용 함수에 비관리자 호출 |
| `InvalidState()` | 현 상태에서 허용되지 않는 호출 |
| `NotEnoughCandidates()` | 후보자 < 2명일 때 `startVoting` |
| `EndTimeInPast()` | `endTime ≤ block.timestamp` |
| `AlreadyVoted()` | 중복 투표 시도 |
| `InvalidCandidate()` | 존재하지 않는 ID 투표 시도 |
| `VotingClosed()` | `block.timestamp ≥ endTime` 이후 vote 호출 |

### 11.1 관리자 인증 방식

별도 로그인 없음. 관리자 API 호출 시 클라이언트가 `X-Wallet-Address`, `X-Signature`, `X-Message` (예: `"admin-action:${nonce}:${timestamp}"`) 헤더를 보내고, 서버가 `ethers.verifyMessage()`로 주소 복원 후 다음 순서로 관리자 검증:

1. **컨트랙트 배포 이력이 있는 경우**: 백엔드가 Sepolia RPC로 현재 컨트랙트의 `owner()` 함수를 호출하여 서명 주소와 비교.
2. **컨트랙트 미배포 상태 (NONE)**: 백엔드 환경변수 `OWNER_ADDRESS`에 사전 등록된 주소와 비교. 이 주소를 가진 지갑만 첫 번째 컨트랙트를 배포할 수 있음.

**Replay Attack 방지**: nonce는 서버에서 발급(`GET /api/nonce?address=0x...`)하고, 사용된 nonce는 DB에 기록(TTL 5분). 동일 nonce 재사용 시 401 반환.

### 11.2 데이터 모델 (SQLite)

```sql
TABLE candidates (
  id          INTEGER PRIMARY KEY,    -- 컨트랙트의 nextCandidateId와 동일 (PATCH로 확정)
  name        TEXT NOT NULL,
  photo_path  TEXT NOT NULL,          -- /uploads/uuid.jpg
  status      TEXT DEFAULT 'pending', -- 'pending' | 'confirmed' | 'deleted'
  is_active   INTEGER DEFAULT 1,      -- 0: 삭제됨 (온체인 isActive와 동기화). status='confirmed'인 경우만 유효
  created_at  TEXT DEFAULT CURRENT_TIMESTAMP
)

TABLE contract_deployments (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  address         TEXT NOT NULL UNIQUE,   -- 배포된 컨트랙트 주소
  title           TEXT NOT NULL,          -- 투표 제목 (S-N에서 입력)
  deployed_by     TEXT NOT NULL,          -- 배포자 지갑 주소
  deployed_at     TEXT DEFAULT CURRENT_TIMESTAMP,
  ended_at        TEXT,                   -- 투표 종료 시각 (ENDED 시 백엔드가 기록)
  end_reason      TEXT,                   -- "manual" | "timeup"
  winner_id       INTEGER,                -- 최다득표 후보 ID (동점 시 최소 ID)
  winner_name     TEXT,                   -- 비정규화 저장 (백엔드 조회 성능)
  total_votes     INTEGER DEFAULT 0,      -- 종료 시 집계
  is_current      INTEGER DEFAULT 1       -- 1: 현재 활성, 0: 이전
)

TABLE nonces (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  nonce       TEXT NOT NULL UNIQUE,
  address     TEXT NOT NULL,
  created_at  TEXT DEFAULT CURRENT_TIMESTAMP,
  used        INTEGER DEFAULT 0           -- 사용 후 1로 표시, TTL 5분 경과 시 cron 삭제
)
```

> `POST /contracts` 또는 `POST /votings` 호출 시 기존 레코드의 `is_current`를 0으로 일괄 업데이트 후 새 레코드를 1로 삽입. `ended_at`, `winner_*`, `total_votes`는 VotingEnded 이벤트를 백엔드 폴링으로 감지하거나, 프론트엔드가 종료 확인 후 `PATCH /votings/current/result` 엔드포인트로 기록.

### 11.3 등록 트랜잭션 흐름 (FR-03)

```
[Admin UI]                [Backend]               [Contract]
    |                         |                       |
    | 1. 사진 업로드(multipart) |                       |
    |------------------------>|                       |
    |                         | 2. 파일 저장            |
    |                         |    DB insert (pending) |
    |                         | 3. addCandidate()      |
    |                         |    tx data 생성 응답    |
    |<------------------------|                       |
    | 4. 메타마스크 서명·전송  |                       |
    |------------------------------------------------>|
    |                         |                       | 5. CandidateAdded(id)
    | 6. tx receipt + id 수신 |                       |
    | 7. PATCH /candidates/:id|                       |
    |    (status→confirmed)   |                       |
    |------------------------>|                       |
```

> 트랜잭션이 실패하면 백엔드의 후보자 레코드는 `status='pending'` 상태로 유지된다. 일정 시간(예: 5분) 내 확정되지 않으면 자동 삭제하는 정리 작업을 cron 또는 서버 부팅 시 1회 수행.

  - `deployment` : **isDeployable** (현재 투표 상태가 ENDED 또는 배포 이력 없음 여부), **canDeploy** (precheck 결과)

## 14. 디자인 톤 & 시각화 컨셉

컨셉: **"한국 지상파 개표방송"**

- **컬러**: 다크 네이비 배경(`#0B1B3B`) + 후보별 강한 식별 컬러(빨강/파랑/초록/노랑 등)
- **타이포**: 거대한 숫자 카운터 (Pretendard / Spoqa Han Sans, 굵은 웨이트)
- **차트**:
  - 가로 막대 차트 (후보 사진 + 이름 + 실시간 카운트)
  - 도넛 차트 (전체 표 분포)
  - 누적 투표 수 큰 카운터 ("총 12표")
  - 진행 시간 / 남은 시간 카운트다운 (HH:MM:SS)
- **모션**: 표가 들어올 때 막대 길이가 부드럽게 증가, 1위 변동 시 깜빡임 강조

## 17. 검증 (Acceptance Criteria)

| ID | 시나리오 | 기대 결과 |
|----|----------|-----------|
| AC-01 | 비관리자가 `addCandidate()` 호출 | 트랜잭션 revert, UI에서 "관리자만 가능합니다" |
| AC-02 | 후보자 1명일 때 `startVoting` | `revert(NotEnoughCandidates)`, UI에서 시작 버튼 비활성 |
| AC-03 | 같은 지갑 2번째 투표 | `revert(AlreadyVoted)`, 사람이 읽을 메시지 표시 |
| AC-04 | 종료 시각 이후 투표 | `revert(VotingClosed)`, UI는 결과 화면으로 자동 전환 |
| AC-05 | 메타마스크 미연결 상태에서 `/vote` | `/connect` 화면으로 리다이렉트 |
| AC-06 | 다른 지갑 5개로 동시에 다른 후보에게 투표 | 모든 표가 정확히 반영, 합계 일치 |
| AC-07 | 투표 진행 중 페이지 새로고침 | 본인 `hasVoted` 상태와 차트가 정확히 복원 |
| AC-08 | 투표 종료 후 페이지 진입 | 자동으로 ENDED 결과 화면 |
| AC-09 | 비-Sepolia 네트워크 접속 | 차단 + 전환 유도 |
| AC-10 | 백엔드 다운 상태에서 결과 조회 | 이름/사진은 placeholder, 표 수는 정상 표시 |
| AC-11 | 투표 ACTIVE 중 `/new` 접근 | 메인(`/`)으로 리다이렉트, "진행 중인 투표가 있습니다" 토스트 |
| AC-12 | 투표 ENDED 후 `/new` 진입 → 제목 입력 → MetaMask 서명 → 배포 완료 | 새 컨트랙트 주소가 `POST /votings`로 백엔드에 등록되고 `/admin/candidates`로 이동 |
| AC-13 | S1-NONE 화면에서 다른 관리자가 새 투표 배포 | 5초 폴링으로 감지하여 S1-IDLE 화면으로 자동 전환 |
| AC-14 | `/new` 진입 시 `canDeploy=false` (precheck 실패) | 배포 버튼 비활성 + 사유 문구 표시 |

### 20.1 Hardhat 컴파일 설정 (`hardhat.config.ts`)

```typescript
import { HardhatUserConfig } from "hardhat/config";
import "@nomicfoundation/hardhat-toolbox";

const config: HardhatUserConfig = {
  solidity: {
    version: "0.8.24",
    settings: {
      optimizer: {
        enabled: true,
        runs: 200,          // 배포 1회용 컨트랙트이므로 200 (deploy 비용 vs call 비용 균형)
      },
      evmVersion: "cancun", // Sepolia는 Cancun 하드포크 적용 (2024-03 이후)
      viaIR: false,         // 단순한 컨트랙트이므로 IR 파이프라인 불필요
    },
  },
  networks: {
    sepolia: {
      url: process.env.SEPOLIA_RPC_URL || "",
      accounts: process.env.DEPLOYER_PRIVATE_KEY
        ? [process.env.DEPLOYER_PRIVATE_KEY]
        : [],
    },
  },
  etherscan: {
    apiKey: process.env.ETHERSCAN_API_KEY || "",
  },
};

export default config;
```

### 20.2 컴파일 결과물 활용

Hardhat 컴파일 후 생성되는 `artifacts/contracts/Voting.sol/Voting.json`에서 ABI와 bytecode를 추출하여 프론트엔드 번들에 포함한다.

```typescript
// frontend/src/contracts/Voting.json 으로 복사 또는 심볼릭 링크
// vite.config.ts에 json import 허용 (기본 지원)
import VotingArtifact from "./contracts/Voting.json";

const abi = VotingArtifact.abi;
const bytecode = VotingArtifact.bytecode;
```

> ⚠️ `bytecode`는 프론트엔드 번들에 포함된다. 실습 환경이므로 무방하나, 프로덕션에서는 백엔드가 배포 트랜잭션을 대신 조립하는 방식을 권장.

### 20.3 optimizer `runs` 값 선택 근거

| runs 값 | 최적화 대상 | 적합한 경우 |
|---------|------------|------------|
| 1 | 배포 비용 최소화 | 1회 배포 후 거의 호출 없는 컨트랙트 |
| **200** | 배포·호출 균형 (기본값) | **본 시스템 — 배포 1회, vote() 수십 회** |
| 10000+ | 호출 비용 최소화 | 수백만 건 호출이 예상되는 DeFi 컨트랙트 |

## 24. 시간 포맷 표준

### 24.1 기본 원칙

| 구분 | 형식 | 설명 |
|------|------|------|
| 온체인 (`endTime`) | Unix timestamp (uint256, 초 단위) | Solidity `block.timestamp`와 동일 단위 |
| 백엔드 DB 저장 | ISO 8601 UTC (`2026-05-05T14:59:00Z`) | SQLite TEXT 컬럼, 비교 연산 가능 |
| API 응답 | ISO 8601 UTC (`2026-05-05T14:59:00Z`) | 프론트가 로컬 타임존으로 변환 |
| 프론트 표시 | KST (`2026-05-05 23:59:00`) | UTC+9 변환 후 표시 |
| 사용자 입력 (종료 일시) | `<input type="datetime-local">` | 브라우저 로컬 타임존 기준 → 서버 전송 전 UTC ISO로 변환 |

### 24.2 타임존 변환 규칙

```typescript
// 프론트엔드: 컨트랙트 endTime(unix) → KST 표시
function formatKST(unixSeconds: number): string {
  return new Date(unixSeconds * 1000).toLocaleString("ko-KR", {
    timeZone: "Asia/Seoul",
    year: "numeric", month: "2-digit", day: "2-digit",
    hour: "2-digit", minute: "2-digit", second: "2-digit",
    hour12: false,
  });
  // 예: "2026. 05. 05. 23:59:00"
}

// 프론트엔드: datetime-local 입력값 → Unix timestamp (컨트랙트 전달용)
function localInputToUnix(localDatetimeStr: string): number {
  // datetime-local 값은 브라우저 로컬타임 기준
  return Math.floor(new Date(localDatetimeStr).getTime() / 1000);
}

// 백엔드: Unix timestamp → ISO 8601 UTC (DB 저장용)
function unixToISO(unixSeconds: number): string {
  return new Date(unixSeconds * 1000).toISOString();
  // 예: "2026-05-05T14:59:00.000Z"
}
```

### 24.3 엣지 케이스 주의사항

- **서머타임(DST)**: 한국(KST, UTC+9)은 서머타임 없음 → 오프셋 고정.
- **블록 타임스탬프 신뢰도**: `block.timestamp`는 채굴자가 약간 조작 가능(±15초). 실습 규모에서는 무시 가능.
- **카운트다운 표시**: 프론트의 클라이언트 시계 기준. 사용자 PC 시간이 틀린 경우 표시가 어긋날 수 있음. 정확도가 중요한 경우 백엔드 `/api/time` 엔드포인트로 기준 시각 동기화.
- **`endTime` 입력 최소값**: UI에서 현재 시각 + 5분 이상만 선택 가능하도록 `<input min="...">` 처리 권장.

- Owner 지갑 연결 → 관리자 메뉴 노출
- 5초 후 상태 자동 갱신 확인

**회귀 테스트**: Phase 3~5 API 정상 응답 유지 확인

### Phase 7 — 프론트엔드 메인 화면 (NONE·IDLE·ACTIVE)

**목표**: 상태에 따라 자동 분기되는 메인 대시보드 완성 (실시간 개표 포함)

| 작업 | 세부 내용 |
|------|----------|
| `Home.tsx` 분기 | `VotingContext.state` 에 따라 NONE→S1-NONE, IDLE→S1-IDLE, ACTIVE→S1-Active, ENDED→S8 렌더링 |
| `S1-NONE` | 과거 투표 목록 3건(`GET /votings`), "새 투표 만들기" CTA (관리자에게만), "전체 보기" → `/history` |
| `S1-IDLE` | 후보자 카드 그리드(`GET /candidates`), 컨트랙트 주소 Etherscan 링크, `VotingStarted` 이벤트 수신 시 ACTIVE 전환 |
| `S1-ACTIVE` | 실시간 가로 막대차트(Recharts `BarChart`), 도넛차트(`PieChart`), 총 득표 카운터, 카운트다운 타이머, "투표하러 가기" CTA |
| `BarChart.tsx` | 후보자 순위 자동 정렬, 1위 강조 컬러, 표 추가 시 막대 애니메이션 |
| `DonutChart.tsx` | Recharts `PieChart`, 각 후보 구분 컬러 |

**검증 기준**
- 상태별 화면 분기 정확히 동작
- `Voted` 이벤트 수신 시 차트 실시간 갱신
- 카운트다운 1초 단위 감소
- `VotingStarted` 이벤트 수신 시 IDLE → ACTIVE 즉시 전환

**사이드 이펙트 확인**: WalletContext 재연결 시 상태 초기화 여부

**회귀 테스트**: Phase 6 라우팅·가드 정상 동작 확인

### Phase 8 — 프론트엔드 관리자: 배포 + 후보자 관리

**목표**: 컨트랙트 배포 흐름(S-N) + 관리자 홈(S2) + 후보자 CRUD(S3) 완성

| 작업 | 세부 내용 |
|------|----------|
| `S-N` (`/new`) | 진입 시 `precheck` 호출. 제목 입력(1~50자). 배포 버튼 → `ContractFactory.deploy()` → 4단계 진행 인디케이터 → `POST /votings` → `/admin/candidates` 이동. Race 처리(3단계 실패 시 컨트랙트 주소 표시) |
| `S2` (`/admin`) | 현재 상태·후보자 수·컨트랙트 주소 표시. 카드 네비게이션(후보자 관리, 투표 제어). "새 배포" 버튼 (ENDED/NONE일 때만 활성) |
| `S3` (`/admin/candidates`) | 사진 업로드(미리보기, 2MB 제한) → `POST /candidates` → MetaMask `addCandidate()` 서명 → `PATCH /candidates/:id`. 수정(이름·사진 백엔드만), 삭제(컨트랙트+백엔드). ACTIVE/ENDED 시 폼 비활성 |
| 서명 헬퍼 | `signAdminMessage(nonce)` — `GET /nonce` → `signer.signMessage()` → 헤더 조립 |

**검증 기준**
- `/new` 진입 → precheck 결과 표시
- 배포 버튼 클릭 → MetaMask 팝업 → 4단계 순차 완료 → `/admin/candidates` 이동
- 후보자 등록 → pending 상태 → 트랜잭션 확정 → confirmed 전환
- ACTIVE 상태 → 등록 폼 비활성 확인

**사이드 이펙트 확인**: 배포 완료 후 `VotingContext` 컨트랙트 주소 갱신 여부

### Phase 9 — 프론트엔드: 투표 제어·투표 화면·완료·내 이력

**목표**: 투표 시작/종료(S4) + 투표 행위(S5·S6) + 내 이력(S7) 완성

| 작업 | 세부 내용 |
|------|----------|
| `S4` (`/admin/control`) | `datetime-local` 입력 → `localInputToUnix()` 변환 → `startVoting(endTime)` MetaMask 서명. 강제 종료 버튼 → 확인 다이얼로그 → `endVoting()`. ENDED 시 폼 전체 비활성. (§24 시간 포맷 표준 준수) |
| `S5` (`/vote`) | 후보자 카드 단일 선택 토글. "투표하기" 버튼 → `vote(id)` 트랜잭션 → 3단계 모달(서명대기·채굴대기·완료). `hasVoted == true` 또는 `state != ACTIVE` 진입 시 결과 화면으로 리다이렉트 |
| `S6` (투표 완료) | tx hash·블록번호·시각 표시. Etherscan 링크. "실시간 결과 보러가기" → `/` |
| `S7` (`/me`) | `getMyVote()` 호출 → 투표 여부·선택 후보·tx hash 표시. 미투표 시 안내 문구 |

**검증 기준**
- `startVoting` 트랜잭션 후 IDLE → ACTIVE 상태 전환 확인
- `endVoting` 후 ACTIVE → ENDED 전환 + S8 화면 이동 확인
- 투표 완료 후 같은 지갑 재진입 시 버튼 비활성 확인
- `/me`에서 tx hash Etherscan 링크 정확성 확인

**사이드 이펙트 확인**: 투표 완료 후 S1 ACTIVE 차트 즉시 갱신 여부

### Phase 10 — 프론트엔드: 최종 결과 화면 + 과거 이력

**목표**: ENDED 상태 결과 표시(S8) + 과거 투표 전체 이력(`/history`) 완성

| 작업 | 세부 내용 |
|------|----------|
| `S8` (`/result`, ENDED) | 1위 후보 강조(왕관 아이콘). 컨페티 애니메이션 1회(`canvas-confetti`). 도넛차트 + 순위표(순위·이름·표·비율). 종료 사유("시간 만료" / "수동 종료") 표시. 컨트랙트·Voted이벤트 Etherscan 링크 |
| 결과 저장 트리거 | ENDED 진입 시 `PATCH /votings/current/result` 호출(한 번만). `result_saved` 로컬 플래그로 중복 호출 방지 |
| 관리자 ENDED 화면 | S8 하단에 "새 투표 만들기 →" 버튼 노출 (관리자만) |
| `/history` | `GET /votings` 전체 목록. 날짜·제목·득표수·당선자 표시. 각 항목 Etherscan 링크 |

**검증 기준**
- ENDED 전환 시 S8 자동 이동 확인
- 컨페티 1회만 실행 확인 (새로고침 시 재실행 없음)
- `PATCH /votings/current/result` 1회만 호출 확인
- `/history` 3건 이상 과거 투표 목록 정상 표시

**사이드 이펙트 확인**: `/result` 직접 접근 시 ACTIVE 상태에서의 동작

**회귀 테스트**: Phase 7~9 메인·관리자·투표 화면 전체 동작 확인

```
Week 1
  Day 1  Phase 1  ── 프로젝트 셋업

  Day 4  Phase 4  ── 백엔드 후보자 API
  Day 5  Phase 5  ── 백엔드 투표·배포 API

Week 2
  Day 6  Phase 6  ── 프론트 기반 (지갑·Context·라우터)
  Day 7  Phase 7  ── 프론트 메인 화면 (NONE·IDLE·ACTIVE)
  Day 8  Phase 8  ── 프론트 관리자 (배포·후보자)
  Day 9  Phase 9  ── 프론트 투표 제어·투표·이력
  Day 10 Phase 10 ── 프론트 결과·과거 이력

Week 3
  Day 11 Phase 11 ── E2E 통합 테스트·버그픽스·README
```

**의존성 그래프**

```
Phase 1 (셋업)
    └── Phase 2 (컨트랙트)
            └── Phase 3 (백엔드 기반)
                    └── Phase 4 (후보자 API)
                            └── Phase 5 (투표 API)
                                    └── Phase 6 (프론트 기반)
                                            ├── Phase 7 (메인 화면)
                                            └── Phase 8 (관리자 화면)
                                                    ├── Phase 9 (투표 화면)
                                                    └── Phase 10 (결과 화면)
                                                            └── Phase 11 (E2E)
```

> 각 Phase는 이전 Phase가 완전히 완료(체크리스트 ⑥ 통과)된 후에만 시작한다. Phase 5 완료 시점에 백엔드 전체를 Postman으로 한 번 더 일괄 검증하는 것을 권장한다.
