# PART 1. PRD (Product Requirements Document)

## 1.1 배경 및 목적

meet.u는 IT 프로젝트 팀 매칭 플랫폼이다. 팀장이 모집 포지션·요구 스킬을 등록하면, AI 매칭 모듈이 후보 회원 풀에서 **상위 3개 팀 조합**을 추천한다.

기존 룰 기반/유사도 기반 매칭과 달리, 본 모듈은 다음 두 가지를 동시에 최적화한다.
1. **인물–직무 매칭 점수** (IFN 기반 직관적 퍼지 평가)
2. **팀 내부 협업 효율** (협업 의지 r_ii' 행렬 기반)

해당 모델은 NP-hard 조합 최적화 문제이며, **유전 프로그래밍(GP) + Q-learning** 하이브리드(RL-GP) 알고리즘으로 해결한다.

## 1.2 핵심 사용자 (페르소나)

| 페르소나 | 시스템 내 역할 | AI 모듈과의 접점 |
|----------|----------------|------------------|
| 팀장 (모집자) | 프로젝트 공고 등록 후 추천 요청 | 추천 결과 3팀을 받음 |
| 일반 회원 (지원자 후보) | 본인 포지션·스킬 등록 | 백그라운드로 IFN 계산이 수행됨 |

## 1.3 핵심 가치 제안

- **품질**: 후보가 적은 환경(회원 30명)에서도 다양성을 확보한 3개 팀 추천
- **속도**: 추천 버튼 클릭 시 사용자 체감 가능한 시간(p95 ≤ 30초) 내 응답
- **재현성**: 동일 입력에 대해 결과 추적 가능 (Q-table/GP 모델 버전 명시)

## 1.4 범위

### 1.4.1 포함 (In-scope)

- IFN 계산 (LLM 기반 자동 산출)
- 추천 요청 시점의 후보군 메모리 필터링
- RL-GP 기반 팀 조합 최적화
- 상위 3팀 다양성 패널티 적용
- GP 모델·Q-table의 파일 기반 관리 (환경변수 + 서버 재시작 방식, PART 5 참조)

### 1.4.2 제외 (Out-of-scope)

- 인증/회원/공고 CRUD UI 및 백엔드 라우트 구현
- DB 테이블 DDL 작성 (스키마는 다른 팀원이 담당)
- 프론트엔드 컴포넌트 구현·디자인
- **CPU 온도(신뢰도) 기능** — AI 미반영. 별도 문서 `cpu_temperature_feature.md` 참조 (백엔드/디자인 팀 담당)
- **지원 요청 알림·수락 플로우** — AI 모듈은 추천 결과(`status: "pending"`)까지만 반환. 알림 발송 → 회원 페이지 수락/거절 UI → `status` 갱신은 모두 백엔드/프론트엔드 담당
- **GitHub API 통합 (v1 제외, v2 이관 확정)** — LLM IFN 산출에 GitHub 활동 데이터를 활용하는 기능. v1 (5/29 데모)에서는 미적용. v2 졸업 보고서 시점에서 검토·구현 (2026-05-14 결정, OQ-09 resolved). 양쪽 doc 통일 완료.

### 1.4.3 전제 조건 (Hard Preconditions)

본 모듈이 정상 동작하기 위해 다음 두 가지를 **시스템 차원의 필수 입력**으로 가정한다.

- **회원 프로필 작성 시 기술 스택 태그 필수 선택** — 빈 프로필 불가
- **팀장 모집글 작성 시 요구 기술 스택 태그 필수 선택** — 빈 모집글 불가

→ 따라서 "사용자가 선택하지 않은 스킬"은 **데이터 부재가 아니라 명시적 "미보유" 의사표현**으로 해석된다. 이 해석에 따라 미선택 스킬은 default IFN으로 채워져 매칭 점수 계산에 사용된다 (상세는 PART 4.3 참조).

## 1.5 사용자 시나리오

### 시나리오 A — 회원의 스킬 등록 (IFN 자동 계산)

1. 회원이 본인 포지션·기술 스택 버튼 선택 후 저장 (프론트엔드 → `PUT /portfolio/`)
2. 백엔드가 AI 모듈의 `POST /ai/ifn/compute` 호출 (요청에 `user_id`만 포함)
3. AI 모듈이 백엔드 `GET /user/{userId}/skills` 호출해 입력 스킬 조회
4. AI 모듈이 IFN `<mu, nu>` 산출
5. AI 모듈이 백엔드 `POST /user/{userId}/skill-ifn` 호출해 결과 저장
6. (사용자에게는 즉시 화면 응답, 계산은 백그라운드)

### 시나리오 B — 팀장의 추천 요청

1. 팀장이 모집글 페이지에서 **"AI 팀 추천"** 버튼 클릭
2. 백엔드가 AI 모듈의 `POST /ai/match/recommend` 호출 (요청에 `project_id` + `excluded_user_ids`)
3. AI 모듈이 백엔드 호출로 데이터 수집:
   - `GET /project/{id}/positions` — 요구 포지션·스킬
   - `GET /user/{userId}/skills` — 후보 스킬 (전 후보)
   - `GET /user/{userId}/skill-ifn` — 후보 IFN (전 후보)
   - `GET /cooperation/{userA}/{userB}` — 협업 의지 (후보 쌍)
4. AI 모듈이 후보군 필터링 (1개 이상 겹침) → RL-GP 실행(100세대) → 상위 3팀 도출
5. AI 모듈이 백엔드 `POST /project/{id}/match-result` 호출해 결과 저장
6. AI 모듈이 호출 측(백엔드)에 성공 응답 + 결과 그룹(`teams[]`)을 함께 반환
7. 백엔드는 응답을 프론트엔드로 전달 → 팀장 화면에 상위 3팀 표시
8. 팀장이 멤버별로 **지원 요청** / **제외** 선택
9. 팀장이 **새로고침** 시 제외 명단 반영 후 재실행

## 1.6 성공 지표 (KPI)

| 지표 | 목표 |
|------|------|
| 추천 응답시간 p95 | ≤ 30초 (회원 30명 풀 기준) |
| 추천 응답시간 p50 | ≤ 15초 |
| IFN 계산 비동기 완료율 | ≥ 99% (1분 내) |
| 3팀 다양성 (멤버 평균 겹침률) | ≤ 50% |
| 추천 → 지원 요청 전환율 (실사용 단계 KPI) | ≥ 20% |
| 추천 실패율 (에러/타임아웃) | ≤ 1% |

# PART 2. 화면 설계서 (AI 관련 화면 기능 명세)

> UI/디자인은 디자인 팀(김효주) 담당. 본 문서는 **AI 결과가 보이는 화면의 기능 흐름**만 명세.
> 화면 ID는 AI 모듈 ↔ 백엔드 협의용 식별자이며, 실제 라우트/컴포넌트명은 프론트 팀 자율.

## 화면 S-01. 회원 스킬 등록 화면

| 항목 | 내용 |
|------|------|
| 트리거 | 마이페이지 → 기술 스택 버튼 선택 후 "저장" |
| AI 모듈 접점 | 백엔드가 AI 모듈에 IFN 계산 요청 |
| 사용자 피드백 | "저장 완료" 즉시 표시. IFN 계산은 백그라운드 비동기 |
| 실패 처리 | IFN 계산 실패 시 사용자에게 노출하지 않음. 백엔드 재시도 큐로 위임 |
| AI 모듈이 화면에 노출하는 정보 | **없음** (IFN 값은 내부 값으로만 사용) |

## 화면 S-02. 모집글 작성 화면 (AI 관점 명세만)

| 항목 | 내용 |
|------|------|
| 트리거 | 팀장이 신규 프로젝트 모집글 등록 |
| AI 모듈 접점 | 등록 완료 시점에는 호출 없음 (등록 자체는 DB만) |
| 화면 표기 | "AI 추천은 등록 후 모집글 상세에서 실행할 수 있습니다" 안내 문구 (선택) |

### S-04 기능 명세

| 액션 | 동작 |
|------|------|
| **[지원 요청]** | 해당 멤버 1명에게만 발송 (팀 단위가 아님). 같은 멤버 중복 발송 차단 |
| **[제외]** | 해당 회원을 다음 새로고침에서 제외 명단에 추가. 즉시 결과 화면이 갱신되지는 않음 |
| **[새로고침]** | 제외 명단을 반영해 AI 모듈 재호출 → 결과 영역 갱신. 동일 세션 내 최대 횟수 제한(권장 5회) |
| **점수 표시** | 매칭 점수는 0~1 범위. 소수점 둘째 자리 표기. 단순 시각 정렬용으로만 사용 |
| **빈 결과** | "추천 가능한 조합이 없습니다" 안내 + 후보 부족 원인 표시 |

### 3.1.1 3-파이프라인 구조

```
┌────────────────┬────────────────────┬──────────────────────────────┐
│  파이프라인 1   │     파이프라인 2     │         파이프라인 3            │
│  유저 역량 분석  │    후보군 필터링      │    RL-GP 기반 팀 구성 최적화      │
│  (백그라운드)   │    (추천 시 실시간)   │      (추천 버튼 클릭)            │
├────────────────┼────────────────────┼──────────────────────────────┤
│ 트리거:         │ 트리거:              │ 트리거:                        │
│ 프로필 저장      │ 추천 버튼 클릭        │ 추천 버튼 클릭                  │
├────────────────┼────────────────────┼──────────────────────────────┤
│ FR-AI-01       │ FR-AI-02            │ FR-AI-03 + FR-AI-04         │
│ 결과: IFN 저장   │ 결과: 메모리 후보군    │ 결과: 상위 3팀 응답             │
└────────────────┴────────────────────┴──────────────────────────────┘
       │                  │                          │
```

- 파이프라인 1·2는 **동일한 포지션·기술 스택 정의표**를 공유한다 (주체만 다름: 유저 vs 팀장)

- 파이프라인 3은 100세대 GP를 **1회만 실행**하여 다양한 팀 조합을 한 번에 수집 (4.10 설계 근거 참조)

## 3.2 기능 요구사항 (Functional Requirements)

### FR-AI-01. IFN 계산 (LLM 기반)

| 항목 | 내용 |
|------|------|
| 설명 | 회원이 **클릭(선택)한 스킬에 한해서만** LLM이 IFN `<mu, nu>` 산출 |
| 트리거 입력 | `{ user_id }` — 백엔드가 트리거 |
| 데이터 수집 | AI 모듈이 백엔드 `GET /user/{userId}/skills` 호출하여 스킬 목록 획득 |
| 처리 | IFN `<mu, nu>` 산출 (스킬당 1행) |
| 결과 저장 | AI 모듈이 백엔드 `POST /user/{userId}/skill-ifn` 호출하여 저장 |
| 응답 | 호출 측에 `{ status: "success", user_id, computed_count }` 반환 |
| 제약 | `mu + nu ≤ 1`, `mu, nu ∈ [0, 1]`, `pi = 1 - mu - nu` (pi는 파생값) |
| LLM | SAIFE X (학교 지원 LLM 거버넌스 플랫폼) — 프롬프트는 모듈 내 템플릿으로 고정 |
| 실패 정책 | LLM 호출 실패 시 5xx 반환. 모듈 자체 재시도는 최대 2회(지수 백오프) |
| 비고 | `evaluated_at`(UTC ISO 8601)로 산출 시점을 명시 — LLM 응답 분포 변화 추적용 |
| **미클릭 스킬 처리** | LLM 호출 없음. 매칭 시점(FR-AI-03/PART 4.3)에서 default IFN `(mu=0, nu=0.7, pi=0.3)` 으로 자동 채움 |

#### IFN 갱신 정책

| 트리거 | 갱신 범위 | 시점 |
|--------|---------|------|
| 회원이 프로필 스킬 태그 추가/제거 | 해당 회원의 전 스킬 | 즉시 (FR-AI-01 호출) |
| 프로젝트 완료 + 평가 종료 | 해당 회원의 사용 스킬 | 평가 기간(14일) 종료 직후 (백엔드 트리거) |
| 정기 안전망 (cron) | 전 활성 회원 | **월 1회** — 매월 1일 03:00 KST. `last_ifn_updated_at`이 30일 이상 지난 회원만 |

→ 정기 cron은 LLM 기반 IFN(최종본)에서 의미 있음. v1의 고정 매핑 단계에서는 cron 미실행 (`USE_MOCK_BACKEND` 또는 `IFN_MODE=fixed`로 분기).

### FR-AI-03. RL-GP 팀 매칭 실행

| 항목 | 내용 |
|------|------|
| 설명 | 후보군과 요구 포지션을 입력으로 RL-GP를 100세대 실행 |

| 출력 | 전 세대에 걸쳐 생성된 모든 (팀 조합, fitness) 수집 → 정렬 → 다양성 패널티 적용 → 상위 3팀 |

| 라이브러리 | DEAP |
| 결정성 | 동일 입력 + 동일 시드 + 동일 모델 버전 → 동일 결과 |

### FR-AI-04. 다양성 패널티 적용

| 항목 | 내용 |
|------|------|
| 설명 | 상위 후보 팀 중 1·2·3위가 멤버 50% 이상 중복되지 않도록 패널티 부여 |
| 규칙 | 포지션 수가 N일 때, `ceil(N/2)`명 이상 겹치면 패널티 (예: N=7일 때 4명 이상) |
| 처리 | 1위 확정 → 2위 후보들 중 1위와 50% 겹침이면 fitness에 페널티 계수 곱한 뒤 재정렬 → 2위 확정 → 동일 절차로 3위 |
| 패널티 계수 | 0.5 (구성 가능, 모듈 설정 파일에 명시) |

**구체 예시 — 포지션 7개 기준**

```
7명 중 4명 이상 겹침 → 57% → 패널티 적용 (fitness × 0.5)
7명 중 3명 이하 겹침 → 43% → 유지 (패널티 없음)
```

**선정 순서 의사코드**

```
all_teams = 100세대에 걸쳐 수집된 모든 (팀, fitness)
ranked    = sort(all_teams, fitness desc)

1위 = ranked[0]            # 최고 fitness 팀 확정
2위 후보 = ranked[1..] 에서 1위와 ⌈N/2⌉명 이상 겹치면 fitness *= 0.5
2위 = 패널티 적용 후 재정렬한 결과의 최상위
3위 후보 = 2위와 동일 절차 — 단, 1위·2위 둘 다와 비교
3위 = 패널티 적용 후 재정렬한 결과의 최상위
```

### 3.3.A AI 모듈이 백엔드에 노출하는 엔드포인트

#### 3.3.A.1 `POST /ai/ifn/compute`

요청 (백엔드 → AI)
```json
{ "user_id": "user_123" }
```

AI 모듈 내부 동작:
1. 백엔드 `GET /user/{user_id}/skills` 호출 → 스킬 목록 조회
2. LLM(SAIFE X)으로 IFN 산출
3. 백엔드 `POST /user/{user_id}/skill-ifn` 호출 → IFN 저장

응답 (200)
```json
{
  "status": "success",
  "user_id": "user_123",
  "computed_count": 3,
  "prompt_version": "ifn-v1.2",
  "evaluated_at": "2026-05-01T09:00:00Z"
}
```

응답 (5xx — LLM 실패)
```json
{ "error_code": "LLM_FAILURE", "message": "IFN 계산 실패" }
```

#### 3.3.A.2 `POST /ai/match/recommend`

요청 (백엔드 → AI)
```json
{
  "project_id": "proj_456",
  "excluded_user_ids": [],
  "seed": 42
}
```

AI 모듈 내부 동작:
1. 백엔드 `GET /project/{project_id}/positions` → 요구 포지션·스킬
2. 백엔드 `GET /project/{project_id}/candidates` → 후보 목록 *(TBD 엔드포인트)*
3. 후보별 `GET /user/{userId}/skill-ifn` → IFN 조회
4. 후보 쌍별 `GET /cooperation/{userA}/{userB}` → 협업 의지 조회
5. 후보 필터링 (1개 이상 겹침) + RL-GP 실행
6. 백엔드 `POST /project/{project_id}/match-result` → 결과 저장
7. 호출 측에 응답 반환

응답 (200) — 결과는 백엔드에 저장 완료 + 호출 측에 화면용 데이터 반환
```json
{
  "status": "success",
  "project_id": "proj_456",
  "model_version": "v1.0",
  "q_table_version": "v1.0",
  "generated_at": "2026-05-01T10:00:00Z",
  "teams": [
    {
      "rank": 1,
      "_objective": 0.873,
      "members": [
        { "position_id": "pos_frontend", "user_id": "user_123", "_match_score": 0.92 },
        { "position_id": "pos_backend",  "user_id": "user_456", "_match_score": 0.85 }
      ]
    }
  ]
}
```

> **점수 노출 정책**: `_objective`·`_match_score` 필드는 **내부용** (`_` 접두사로 표시). 백엔드의 정렬·감사·로깅에는 사용하되 **프론트엔드 화면에는 표시하지 않는다**. 백엔드는 프론트엔드로 전달 시 해당 필드를 제거해야 함.
> `teams`는 프론트엔드가 화면 렌더링에 쓰는 그룹 단위 포맷 (순위·멤버 정보만 사용자에게 노출).
> 백엔드 `POST /project/{id}/match-result` 호출 시 전송되는 행 단위 포맷은 3.3.B 절 참조.
> `status` 초기값은 항상 `"pending"` — 사용자가 지원 요청을 보내거나 거절·확정될 때 백엔드가 갱신.

응답 (422 — 후보 부족)
```json
{ "error_code": "INSUFFICIENT_CANDIDATES", "message": "후보 수가 포지션 수보다 적습니다." }
```

응답 (504 — 타임아웃)
```json
{
  "error_code": "MATCHING_TIMEOUT",
  "message": "분석 시간이 초과되었습니다.",
  "request_id": "req-7f3c...",
  "elapsed_ms": 60123
}
```
→ 프론트엔드는 이 응답을 받으면 S-06 매칭 실패 화면을 표시.

### 3.3.B AI 모듈이 백엔드에 호출하는 엔드포인트 (의존성)

> 아래 엔드포인트는 **백엔드 측이 제공해야 한다**. 현재 백엔드 Swagger에 미등록 — 최현정님이 추가 필요.

| Method | Endpoint | 호출 시점 | 용도 |
|--------|----------|-----------|------|
| GET | `/user/{userId}/skills` | FR-AI-01, FR-AI-02 | 유저 기술 스택 조회 |

# PART 4. 알고리즘 명세 (RL-GP)

> 본 섹션은 SRS의 FR-AI-03을 구현하기 위한 알고리즘 세부 명세이다.

## 4.1 결정 변수 및 목적 함수

**결정 변수**: `x_ij ∈ {0,1}` — 후보 i를 포지션 j에 배정하면 1.

**목적 함수**:
```
max  F = Σᵢ∈C Σⱼ∈POS ezᵢⱼ · xᵢⱼ · γ

  ezᵢⱼ : IFN 기반 인물–직무 매칭 점수
  γ    : 팀 협업 효율 (v1 범위 [0.80, 1.10], pair factor 평균 + clamp — 아래 γ 계산 참조)
```

**γ 계산** (v1 MVP — pair factor 평균 + clamp 방식, 2026-05-14 팀 결정):
```
pair_factor(i, j):
  r_ii' = +1  →  1.05  (긍정 협업 이력)
  r_ii' =  0  →  1.00  (이력 없음 또는 중립)
  r_ii' = -1  →  0.90  (부정 협업 이력)

communication_efficiency_gamma(team)
  = mean( pair_factor(i, j) for all (i, j) pairs in team )
  → clamp to [0.80, 1.10]

단일 사용자 팀 또는 계산 가능한 쌍이 없으면 γ = 1.00
```

> **결정 근거**: 논문식 `γ = ½(3+Σr·x·x/[Σn(s)]²)` (범위 [1,2])는 협업 이력이 최종 점수를 최대 2배까지 확장하여 후보-포지션 적합도(ezᵢⱼ)를 협업 점수가 지배할 위험이 있다. v1 MVP에서는 점수 안정성·UX 우선을 위해 pair factor 평균 + clamp [0.80, 1.10] 채택.
>
> **v2 검토 메모**: 협업 데이터가 풍부해지고 논문 정합성 검증이 필요한 시점(졸업 보고서)에 **논문식 γ ∈ [1, 2] 도입 비교 실험 가능**. 옵션 B (사전학습 tree)와 함께 실험 항목으로 검토.

## 4.2 제약 조건

> **meet.u 적용 차이**: 논문의 C5(`RS(posⱼ) ⊆ S(cᵢ)`)는 hard constraint이지만, 회원 30명 규모와 "1개 이상 겹침" 필터(FR-AI-02)와의 정합성을 위해 **soft constraint로 완화**한다. 후보의 미보유 스킬은 default IFN `(0, 0.7, 0.3)`으로 채워져 ezᵢⱼ 계산 시 자동 페널티가 반영되며, 이로써 hard 제약과 동등한 효과를 점수 단계에서 달성한다(PART 4.3 참조).

| 번호 | 조건 | 유형 |
|------|------|------|
| C1 | 각 후보자는 최대 1개 포지션 (`Σⱼ xᵢⱼ ≤ 1`) | hard |
| C2 | 각 포지션은 정확히 1명 (`Σᵢ xᵢⱼ = 1`) | hard |
| C3 | 각 필수 스킬에 요구 인원 수 충족 | hard |
| C4 | 선발된 후보자들의 스킬 합집합 ⊇ 팀 요구 스킬 | hard |
| C5 | 후보가 배정 포지션의 스킬 요건 충족 — 미보유 시 default IFN 페널티로 자동 반영 | **soft** (논문은 hard) |
| C6 | `xᵢⱼ ∈ {0,1}` | hard |

## 4.3 IFN 매칭 점수 계산 (ezᵢⱼ)

### 4.3.1 IFN 채움 (Default Fill) 정책

포지션 j의 요구 스킬 집합 `RS(posⱼ)`에 대해, 각 스킬 차원 l마다 후보 i의 IFN `αᵢⱼₗ`를 다음 규칙으로 확정한다.

| 케이스 | IFN `αᵢⱼₗ = <μ, ν, π>` | 근거 |
|--------|-------------------------|------|
| 후보 i가 스킬 l을 **클릭(선택)함** | LLM 산출 연속값 (FR-AI-01 결과 그대로) | 정상 평가 |
| 후보 i가 스킬 l을 **클릭하지 않음** | **`(μ=0, ν=0.7, π=0.3)`** (고정) | 전제 1.4.3에 따른 명시적 미보유 의사표현. π=0.3은 LLM 모델·사용자 입력 누락에 대한 보수적 마진 |

> 주의: 본 모듈은 USER_SKILL_IFN을 백엔드에서 전달받지만, **누락된 차원은 매칭 시점에 모듈이 자체적으로 default fill** 한다(백엔드는 빈 IFN을 보내도 됨).

### 4.3.2 IFPIWA 집계

모든 차원에 IFN이 확정된 상태에서 다음 IFPIWA 연산자로 가중 집계해 ezᵢⱼ 산출 (논문 식 그대로).

```
d(α, α')   = ½ (|μ - μ'| + |ν - ν'|)
sup(α, α') = 1 - d(α, α')
T(αᵢⱼₗ)    = Σ wⱼₗ' · sup(αᵢⱼₗ', αᵢⱼₗ)
ρᵢⱼₗ       = wⱼₗ(1 + T) / Σ wⱼₗ(1 + T)
IFPIWA     = ⟨ 1 - Π(1 - μ)^ρ ,  Π(1 - μ)^ρ - Π(1 - (μ + ν))^ρ ⟩
```

→ 결과: 미클릭 스킬이 많은 후보일수록 μ 합이 작고 ν 합이 커서 ezᵢⱼ가 자연스럽게 낮아진다. 이로써 C5(soft)가 점수 단계에서 자동 강제된다.

### 4.3.3 스킬 가중치 정책 (`wⱼₗ`)

IFPIWA 수식의 `wⱼₗ`은 포지션 j에서 스킬 l이 얼마나 중요한지를 나타낸다. AI 팀이 도메인 지식으로 사전 정의하며, `ai_module/config/position_skill_weights.yaml` 에 저장한다.

## 4.5 GP 개체 표현

**터미널 셋** (12개): NPT, RNP, SC, SCN, ANS, NSR, NCP, NCR, SCW, SMP, WEC, RAND

**함수 셋**: `{ +, −, ×, ÷, sin, cos, max, min }` (÷는 보호된 나눗셈)

**개체 = 트리** → 후보자별 우선순위 함수.

**디코딩**:
1. 모든 후보에 대해 트리 평가 → 우선순위 점수
2. 최고 점수 후보 1명 선택
3. IFN으로 (해당 후보, 남은 포지션) 중 최적 매칭 → 배정
4. 후보·포지션 제거 후 반복

## 4.6 Q-learning 통합

**상태**: 2가지 (Q-table 파일에서 키 이름)
- `improved` (S0) = 직전 세대 최적 적합도 향상됨
- `no_improve` (S1) = 직전 세대 최적 적합도 미향상

**행동**: 4가지 모드 (P1, P2, P3, P4)

| 모드 | 집단 크기 | 설명 |
|------|-----------|------|
| P1 | N=100 | 표준 GP |
| P2 | N=100 | 전역 최적을 P1의 임의 개체와 교체 |
| P3 | 2 | 전역 최적 + 직전 세대 최적 (지역 탐색) |
| P4 | N+2 | P1 ∪ P3 |

**보상**: `Rₜ = f*ₜ - f*ₜ₋₁`

**Q-table 업데이트**:
```
Q(S, A) ← Q(S, A) + α [R + γ_rl · max_a Q(S', a) - Q(S, A)]
α = 0.01, γ_rl = 0.9
```

**ε-greedy**:
```
if rand() < ε:   a = uniform_random(P1..P4)
else:            a = argmax_a Q(S, a)
```
ε 스케줄: 초기 0.3 → 100세대까지 선형 감쇠 0.05

## 4.7 메인 루프 의사코드

```
init population P (size 100, ramped half-and-half)
init Q-table = zeros(2 x 4)
best_global  = None
last_best    = None

for g in 1..100:
    S = S0 if (current_best_fitness > last_best_fitness) else S1
    A = epsilon_greedy(Q, S, eps_schedule[g])

    P = apply_mode(A, P, best_global, last_best)
    P' = crossover(P, p=0.9) -> mutation(P', p=0.1)

    eval_fitness(P')                       # 100세대 모두 실제 평가 (대리 모델 미사용)
    update best_global, last_best
    R = best_global_fitness - prev_best_global_fitness
    Q[S, A] += alpha * (R + gamma_rl * max(Q[S']) - Q[S, A])

    record_all_teams(P')                   # 세대별 모든 (팀, fitness) 저장

# 상위 3팀 도출
ranked = sort(all_teams, key=fitness, desc)
top3   = select_with_diversity_penalty(ranked, threshold=0.5)
return top3
```

## 4.8 대리 모델 (KNN) 미사용 결정

논문 대비 변경: 회원 30명 / 후보군 ≤ 100명 규모에서 실제 평가 비용이 크지 않다. 대리 모델은 학습/유지 비용 대비 효익이 낮아 **제외**한다. 향후 회원 수 증가 시 재검토.

## 4.9 합성 데이터(Synthetic) 사전 학습 정책 — **졸업 보고서 실험 항목 (MVP 미포함)**

> ⚠️ **중요 — 본 절은 MVP 구현 범위가 아님**
>
> - **본 절의 모든 내용은 졸업 보고서·논문 정합성 검증용 실험 항목이다.**
> - **MVP (서비스 운영) 구현에는 포함되지 않는다.**
> - **서비스 PRD(`developer-team-matching-ai-prd-srs.md`)에는 의도적으로 제외되어 있다.**
> - **v1 sprint (`ai_module_v1_sprint.md`) 5.3에서도 명시적으로 제외.**
> - 본 절은 졸업 보고서 작성 시점에 **논문 RL-GP의 학습 단계**를 실험으로 재현하기 위한 참고 명세이며, 운영 코드에 통합되지 않는다.
> - `trained_on: "synthetic"` 표기는 **실험용 모델 파일**에만 적용되며, 운영 모델은 별도.

회원 30명 환경에서는 실데이터로 GP·Q-table을 학습할 만큼의 표본이 확보되지 않는다. 졸업 보고서에서 논문 RL-GP의 학습 효과를 검증하려면 합성 데이터 사전 학습이 필요하다.

| 항목 | 정책 |
|------|------|
| 사전 학습 데이터 | **합성 시나리오** — 다양한 후보 풀·포지션 조합·r_ii' 행렬을 임의 생성해 학습 인스턴스 구성 |
| 학습 시점 | 배포 전 1회 학습 → `.pkl` / Q-table 파일 산출 → 서비스에 정적 로드 |
| 모델 메타데이터 | `trained_on: "synthetic"` 으로 명시 (5.2/5.3 참조) |
| 운영 중 추가 학습 | 미수행. 회원·평가 데이터 누적 후 재학습 검토 |
| 재현성 | 합성 데이터 생성 시드를 모듈 설정에 명시. 동일 시드 → 동일 학습 결과 |
| **v1 대응** | 본 절 미적용. 매 요청 시 GP를 즉석 진화 (`ai_module_v1_sprint.md` 5.3 참조) |

## 4.10 RL-GP 실행 설계 근거 (옵션 A + 1회 실행 — 2026-05-13 팀 결정)

> **결정 1 — 운영 방식**: **옵션 A (per-request RL-GP 실행)** 확정. 옵션 B (사전 학습 tree 재사용)는 졸업 보고서 작성 시점의 부가 실험 항목으로만 다루며, **MVP 구현에서는 제외**.
>
> **결정 2 — 1회 실행**: 추천 1회 처리 시 RL-GP를 1번만 돌리고, 누적된 모든 (팀, fitness)에서 다양성 패널티로 상위 3팀 도출.
>
> 본 doc은 옵션 A 확정으로 통일. 서비스 PRD(`developer-team-matching-ai-prd-srs.md` 6.4)도 **옵션 A 채택으로 양쪽 doc 통일 완료** (2026-05-14).

상위 3팀을 뽑기 위해 RL-GP를 3번 실행하지 않고 **1번만 실행한 뒤 누적된 모든 (팀, fitness)에서 다양성 패널티로 3개를 도출**하는 방식을 택했다. 근거는 다음과 같다.

| 장점 | 설명 |
|------|------|
| Q-table 학습 효과 극대화 | 100세대 동안 누적된 학습이 그대로 유지됨. 매 실행마다 Q-table을 초기화하는 비용·낭비 제거 |
| 실행 시간 단축 | 단일 RL-GP 실행이면 NFR-PERF-01(평균 30초) 만족 가능. 3회 실행은 시간·자원 모두 3배 |
| 논문 구조 보존 | 논문은 단일 팀 최적화 알고리즘. 1회 실행 + 사후 다양성 선정으로 알고리즘 구조 변경 없이 3팀 추천 확장 |
| 자연스러운 다양성 | 100세대 × 100 집단 = 10,000 후보 팀이 자연 발생. 다양성 패널티(FR-AI-04)로 3개 선정만 하면 됨 |
| 결정성 확보 용이 | 시드 고정 1회만 신경 쓰면 됨. 다중 실행 시 시드 관리 복잡도 증가 |

## 5.2 GP 모델 메타데이터

```json
{
  "version": "v1.0",
  "trained_at": "2026-05-01T00:00:00Z",
  "file_path": "/models/best_tree_v1.pkl",
  "trained_on": "synthetic"
}
```

| 필드 | 의미 |
|------|------|
| `version` | 시멘틱 버전 |
| `trained_at` | 모델 학습 완료 시각 (UTC ISO 8601) |
| `file_path` | `.pkl` 실제 경로 |
| `trained_on` | 학습 데이터 종류 — `"synthetic"` (합성), 향후 `"real"` 또는 `"mixed"` 가능 |

## 5.3 Q-table 구조

Q-table은 별도 파일(JSON)로 저장하며, 상태 2종 × 행동 4종 구조이다.

```json
{
  "version": "v1.0",
  "trained_on": "synthetic_scenarios",
  "q_table": {
    "improved":   { "P1": 0.0, "P2": 0.0, "P3": 0.0, "P4": 0.0 },
    "no_improve": { "P1": 0.0, "P2": 0.0, "P3": 0.0, "P4": 0.0 }
  }
}
```

| 상태 키 | 의미 |
|---------|------|
| `improved` | 직전 세대 최적 적합도 향상됨 (PART 4.6의 S0) |
| `no_improve` | 직전 세대 최적 적합도 미향상 (PART 4.6의 S1) |

| 행동 키 | 의미 |
|---------|------|
| `P1`~`P4` | 4가지 탐색 모드 (PART 4.6 참조) |

> 응답·로그에는 `model_version`, `q_table_version` 만 포함하여 추적 가능. SHA256·호환 버전 검증 등은 적용하지 않음.

## 5.4 운영 점검 가이드

| 상황 | 대응 |
|------|------|
| 추천 응답시간이 평균 30초를 자주 넘김 | 로그 확인 → GP 파라미터(세대 수·집단 크기) 축소 검토 |

# PART 6. 제약·가정·위험

## 6.1 가정

- **회원 프로필·모집글의 기술 스택 태그는 시스템 차원에서 필수 입력으로 강제된다** (1.4.3 전제). 따라서 "미선택 스킬"은 데이터 부재가 아닌 명시적 미보유 의사표현이다.
- 후보 회원의 `USER_SKILL_IFN`은 추천 요청 시점에 백엔드 DB에 적재되어 있다 (AI 모듈이 GET으로 조회). 사용자가 클릭하지 않은 스킬은 IFN이 비어 있어도 무방하며, 매칭 단계(4.3.1)에서 default IFN `(0, 0.7, 0.3)`으로 자동 채워진다.
- **AI 모듈은 백엔드(Node.js) REST 엔드포인트를 호출해 데이터를 수집한다** (3.3.B 참조). DB 직접 접근 금지.
- 백엔드는 3.3.B에 명시된 모든 의존 엔드포인트를 제공한다 (현재 일부는 신설 예정 — `GET /project/{id}/candidates` 등).
- **평가 기간은 프로젝트 완료 처리 시점부터 14일**이다. 14일 이내 미평가는 0점(중립) 자동 처리. 평가 종료 후 백엔드가 `r_ii'`를 갱신하며, AI 모듈은 매 매칭 요청 시 `GET /cooperation/{userA}/{userB}` 로 최신 값을 조회한다.
- 회원 30명 / 동시 30명 환경에서 단일 인스턴스로 충분하다.

## 6.2 제약

- DB 직접 접근 금지
- LLM 호출 결과는 외부 의존이며 변동 가능 → 프롬프트 버저닝 필수
- 모델 파일(.pkl)은 신뢰된 운영자만 업로드

## 6.3 위험 및 완화

| 위험 | 영향 | 완화 |
|------|------|------|
| LLM 응답 지연 | IFN 계산 지연 | 비동기 처리, 2회 재시도, 사용자 화면은 즉시 응답 |
| 후보 < 포지션 수 | 추천 불가 | 422 즉시 반환, 화면 안내 |
| GP 수렴 실패 (모든 세대 동일 best) | 추천 품질 저하 | ε 스케줄 조정, 다음 빌드에서 재학습 |
| RL-GP 무한루프 / 비정상 지연 | 사용자 무한 대기 | `asyncio.wait_for(60)` 으로 60초 제한 + 백엔드 90초 read timeout 이중 안전망 (NFR-PERF-04) |
| 협업 의지 이력 부재 | r_ii' 대부분 0 → γ 효과 약화 | 초기에는 자연스러운 현상으로 수용. 회원·평가 누적 후 자동 개선 |
| 결과 결정성 깨짐 | 동일 입력 다른 결과 | 시드 고정 + 모델 버전 명시 + 회귀 테스트 |
| 백엔드 호출 실패 (GET 데이터 조회) | AI 데이터 수집 실패 → 매칭 불가 | 각 호출 per-call 5s 타임아웃 + 5xx 재시도 2회. 후보 일부 실패는 해당 후보만 제외 |
| 백엔드 호출 누적 지연 | 추천 응답시간 증가 (Pull 방식 다회 호출) | 후보별 호출은 `asyncio.gather` 로 병렬화. 30명 풀 기준 추가 1~3초 예상 |
| 백엔드 엔드포인트 미구현 | 통합 실패 | 3.3.B 표 엔드포인트가 5/16 전에 준비되어야 함. 특히 `/project/{id}/candidates` 신설 합의 필요 |

## 부록 D. 논문 필수 조건 충족 체크리스트

논문이 RL-GP 알고리즘이 정상 동작하기 위해 가정하는 조건과 meet.u에서의 충족 방식.

| 논문 조건 | meet.u 적용 | 충족 |
|----------|-------------|------|
